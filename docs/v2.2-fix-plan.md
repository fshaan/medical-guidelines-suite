> 创建日期: 2026-03-24
> 更新日期: 2026-03-25
> 状态: 待实施

## 问题清单

| # | 问题 | 严重度 | 根因 |
|---|------|--------|------|
| 1 | SKILL.md 无约束禁止 subagent，LLM 自行用并行代理 | 高 | 缺少显式禁令 |
| 2 | 非 Claude 模型无法执行工作流（路径迷失、不调脚本、输出位置不稳定） | 高 | 中间环节无确定性编排 |
| 3 | LLM 承诺不用 subagent 但执行时偏离 | 高 | 文本约束力不足 |
| 4 | 缺少 OpenCode / Gemini CLI 安装适配 | 中 | 只配了 Claude Code + OpenClaw |
| 5 | 检索偏向 NCCN/CSCO，忽略其他组织；患者临床特征（合并症、转移部位、分期等）未被充分纳入检索 | 高 | LLM 走捷径 + 缺全特征触发机制 |
| 6 | 输出质量与患者数量呈反比 | 高 | 上下文窗口污染 |

**核心设计目标**：
- 提高不同模型输出结果的均质性，降低 LLM 智能水平对输出质量的影响
- 映射表由系统从知识库自动生成，不要让用户配置
- 所有患者提供的信息都要被纳入检索条件，不仅仅是合并症
- 输出种类允许用户自定义，输出按患者姓名音序排列

---

## 方案总览

```
当前:  parse → split → [LLM 自由编排] → merge → validate → generate
目标:  parse → orchestrate → [LLM 按预生成 prompt 执行] → merge → validate → generate
               ↑                    ↑
       自动扫描知识库生成映射     脚本提取基础关键词 + LLM 补充
       + 从患者数据提取特征      + 显式上下文重置指令
       + 生成全覆盖 grep 命令
```

**orchestrate 解决的问题**：
- 问题 1/3：脚本顺序执行，无并行可能 + HARD_CONSTRAINT 双层防御
- 问题 2：所有路径预解析为绝对路径，grep 命令预生成
- 问题 5：强制为每个患者生成覆盖**所有 org × 所有临床特征维度**的 grep 命令，LLM 无法跳过
- 问题 6：每个 batch prompt 自包含 + 显式 `<CONTEXT_RESET>` 指令

## Phase 1: batch_pipeline.py — orchestrate 子命令

### 1.1 `resolve_kb_root(explicit_path)` — 知识库路径解析

按优先级：`--kb-root` > `MEDICAL_GUIDELINES_DIR` 环境变量 > `./guidelines/` > `./knowledge/`
验证：目标路径下存在 `data_structure.md`

### 1.2 `scan_knowledge_base(kb_root)` — 自动扫描知识库（核心新增）

**从 data_structure.md 自动提取映射信息，不需要用户配置任何文件。**

扫描流程：
1. 解析根目录 `data_structure.md`：
   - 提取组织列表（从"指南目录概览"表或子目录枚举）
   - 提取"临床问题→指南映射矩阵"（根索引第 112-164 行已有结构化表格）
   - 提取证据等级对照表
2. 遍历每个 org 子目录的 `data_structure.md`：
   - 提取文件清单（文件名含疾病名称，如 `NCCN_GastricCancer_*` → 胃癌）
   - 提取章节索引（章节代码 + 临床问题覆盖）
   - 提取检索关键词建议（各 org 的 data_structure.md 已有此列表）
3. 验证 `extracted/` 目录存在且有 `.txt` 文件

输出：内存中的 `kb_profile` 字典，包含：
```python
{
    "orgs": ["NCCN", "ESMO", "CSCO", "Japanese", "Korean", "CACA"],
    "org_files": {
        "NCCN": [
            {"file": "NCCN_GastricCancer_2026.V2_EN.txt", "disease_hint": "胃癌", "lines": 6497},
            {"file": "NCCN_Esophageal_and_EGJ_Cancers_2026.V2_EN.txt", "disease_hint": "食管癌/EGJ", "lines": 8660}
        ], ...
    },
    "org_chapters": {
        "NCCN": [
            {"code": "GAST-1", "name": "Workup and Additional Evaluation", "topics": "诊断评估、分期"},
            ...
        ], ...
    },
    "org_keywords": {
        "NCCN": {"手术": [...], "化疗": [...], "靶向": [...]},
        ...
    },
    "clinical_question_map": {
        "手术术式 / 淋巴结清扫": {"primary": ["Japanese", "NCCN"], "supplementary": ["Korean", "CSCO"]},
        "围手术期化疗": {"primary": ["NCCN", "ESMO"], "supplementary": ["CSCO", "CACA"]},
        ...
    },
    "root_index_content": "data_structure.md 全文"
}
```

**关键**：这个扫描过程纯粹是解析 markdown 表格，不需要 LLM。当用户添加新指南或新领域时，只要 `data_structure.md` 按模板格式编写，orchestrate 就能自动识别。

### 1.3 `extract_patient_features(patient)` — 全特征提取

**从患者所有 26 个字段中规则化提取搜索关键词，不仅限于合并症。**

按临床维度分组提取：

| 维度 | 来源字段 | 提取规则 | 示例输出 |
|------|---------|---------|---------|
| 诊断 | primary_site, pathology | 直接提取+翻译 | ["胃癌","gastric","adenocarcinoma"] |
| 分期 | staging_prefix, t/n/m_stage | 拼接+关键词 | ["ycT4a","N1","M1","stage IV"] |
| 转移 | m_sites, t4b_invasion | 分割+翻译 | ["腹膜","peritoneal","liver","肝"] |
| 分子 | biopsy_molecular, gross_molecular | 正则解析hj_前缀 | ["pMMR","MSS","PD-L1","CPS"] |
| 治疗 | patient_type, prior_treatment, response | 提取方案名+状态 | ["SOX","PD-1","PD","二线"] |
| 标记物 | abnormal_markers, marker_change | 分割+动态 | ["CEA","CA199","升高"] |
| 临床事件 | tumor_emergency, symptom_change | 匹配关键词 | ["出血","bleeding","急症"] |
| 合并症 | comorbidities | 正则扫描 | ["糖尿病","肾功能不全","elderly"] |
| 特殊 | siewert_type, age | 条件判断 | ["Siewert II","高龄","elderly"] |

**对 narrative 格式**：使用正则从 `clinical_narrative` 全文扫描上述所有维度的关键词。

返回结构：
```python
{
    "diagnosis_keywords": [...],
    "staging_keywords": [...],
    "metastasis_keywords": [...],
    "molecular_keywords": [...],
    "treatment_keywords": [...],
    "marker_keywords": [...],
    "event_keywords": [...],
    "comorbidity_keywords": [...],
    "special_keywords": [...],
    "all_keywords": [...]  # 去重合并
}
```

### 1.4 `generate_grep_commands(patient_features, kb_profile, kb_root)` — 生成 grep 命令

根据特征和知识库扫描结果，为每个患者生成覆盖**所有相关 org**的 grep 命令：

1. 从 `kb_profile.org_files` 确定每个 org 的 `extracted/*.txt` 路径
2. 对每个 org × 每个关键词维度 → 生成 grep 命令
3. 命令使用绝对路径
4. 合并同维度关键词到单条 grep（用 `\|` 分隔）避免命令爆炸
5. 每个 org 标记"必须"（全部 org 都是必须的）

### 1.5 `generate_batch_prompt(batch, kb_profile, batch_idx, total)` — 生成自包含 prompt

每个 `batch_NNN_prompt.md` 包含：

```markdown
# 批次 NNN/MMM 检索任务

<CONTEXT_RESET>
请忽略此消息之前的所有检索结果和患者信息。
以下是一个全新的、独立的批次任务，从零开始处理。
不得引用或参考任何其他批次的结果。
不得使用"同上"、"与前面类似"、"参考前述"等表述。
</CONTEXT_RESET>

<MANDATORY_RULES>
1. 必须逐条执行以下所有 grep 命令，不得跳过任何组织
2. 每位患者的每个指南组织都必须有检索结果
3. 如果某指南未涉及该问题，记录: "该指南未涉及此临床问题"
4. 输出必须为简体中文
5. 可以补充脚本未生成的关键词，但不得删减已有的 grep 命令
</MANDATORY_RULES>

## 知识库
路径: {绝对路径}
根索引:
---
{data_structure.md 全文嵌入}
---

## 患者列表（本批次 K 人）

### 患者 1: {name} ({id})
- 完整临床信息: {所有非空字段展示}
- 脚本推断的疾病类型: 胃癌 (confidence: high/low)
- 脚本推断的临床问题: "{question}"（LLM 可修正）
- 脚本提取的关键词: {按维度分组列出}

#### 必须执行的 grep 命令（共 N 条，不得跳过）
**NCCN（必须）:**
1. grep -n -i "HER2\|MSS\|pMMR" /abs/path/NCCN/extracted/*.txt
2. grep -n -i "first.line\|一线\|perioperative" /abs/path/NCCN/extracted/*.txt
3. grep -n -i "peritoneal\|腹膜\|liver\|肝" /abs/path/NCCN/extracted/*.txt
...
**ESMO（必须）:** ...
**CSCO（必须）:** ...
**Japanese（必须）:** ...
**Korean（必须）:** ...
**CACA（必须）:** ...

#### LLM 补充检索空间
如果在上述 grep 结果中发现需要进一步深入的线索，
可补充执行额外的 grep 命令（限同一知识库范围内）。

### 患者 2: ...

## 输出要求
- 文件路径: {绝对路径}/rag_batch_NNN.json
- JSON Schema: {完整 schema}
- 每个 guideline_results 条目必须包含: guideline, version, recommendation, evidence_level, source_file, source_lines
```

### 1.6 `cmd_orchestrate(args)` — 入口函数

```bash
python scripts/batch_pipeline.py orchestrate \
  --patients Output/patients.json \
  --kb-root /path/to/guidelines \
  --output-dir Output/batches \
  --batch-size 5
```

执行流程：
1. `resolve_kb_root()` → 解析知识库路径
2. `scan_knowledge_base()` → 自动扫描知识库结构（不需要配置文件）
3. 加载 patients.json
4. 对每个患者：`extract_patient_features()` → 全特征提取
5. 对每个患者：`generate_grep_commands()` → 生成 grep 命令
6. 按 batch-size 分批
7. 检测已有 checkpoint（rag_batch_NNN.json）
8. 为每个未完成批次：`generate_batch_prompt()` → 写入 prompt 文件
9. 输出 `orchestration_plan.json`（全绝对路径 + 后续命令 + 统计）

### 1.7 修改 `cmd_generate` — 增强参数

- `--template-dir`：可指定模板路径（默认回退到脚本同级 templates/）
- `--output-dir`：统一所有输出位置（默认 `Output/`）
- `--format`：已有，支持选择性输出（xlsx/docx/pptx/all）
- **输出排序**：DOCX 和 XLSX 中的患者按姓名拼音音序排列（`pypinyin` 或 `locale.strxfrm`）

### 1.8 增强 `cmd_validate` — 全特征覆盖率检查

新增检查项：
- **组织覆盖率**：每位患者的 `guideline_results` 是否覆盖了知识库中所有 org
- **特征覆盖**：检查患者的关键临床特征（转移部位、分子分型等）是否在检索结果中有对应内容
- 覆盖率低于阈值时报 WARNING

## Phase 2: SKILL.md — 约束强化与工作流改写

### 2.1 新增 HARD_CONSTRAINT：执行模式约束

```markdown
<HARD_CONSTRAINT>
### Execution Mode Constraints

1. **禁止 subagent / 并行代理**: 所有批处理步骤在当前会话中顺序执行。
   禁止使用 Agent tool、Task tool 或任何并行机制。
   原因: subagent 无法继承知识库上下文，搜索质量无法保证一致性。

2. **禁止自行编码替代脚本**: 必须使用 scripts/batch_pipeline.py 的子命令。
   原因: 脚本已验证，自行编码引入格式差异和路径错误。

3. **所有输出路径以 orchestration_plan.json 为准**: 不自行决定文件存储位置。

4. **禁止跳过 grep 命令**: orchestrate 生成的命令必须全部执行。
   可以补充额外命令，但不得删减已有的。
</HARD_CONSTRAINT>
```

### 2.2 新增 HARD_CONSTRAINT：全覆盖检索 + 全特征纳入

```markdown
<HARD_CONSTRAINT>
### Full-Coverage Retrieval Rules

1. **所有 org 必须检索**: 不得因"已找到足够信息"提前停止。
2. **所有患者临床特征必须纳入检索**: 不仅合并症，转移部位、分子分型、
   分期、治疗史、肿瘤急症等所有非空字段都应体现在检索中。
3. **不得以"类似"跳过**: 即使两位患者病情相似，也独立执行全部命令。
4. **检索深度一致**: 第 1 位和最后 1 位患者的检索深度必须相同。
</HARD_CONSTRAINT>
```

### 2.3 改写 Step 3A + 3B 统一为 orchestrate 驱动

**Step 3A（N ≤ 5）和 Step 3B（N > 5）统一使用 orchestrate**，差别仅在 batch 数量：

```markdown
#### Step 3: Execute Retrieval via orchestrate

python scripts/batch_pipeline.py orchestrate \
  --patients Output/patients.json --kb-root $KB_ROOT \
  --output-dir Output/batches --batch-size 5

读取 orchestration_plan.json，报告用户。

对 pending_batches 中每个批次（严格顺序）：
1. 读取 batch_NNN_prompt.md
   — 注意开头的 <CONTEXT_RESET> 指令，清除前批上下文
2. 按 prompt 中的 grep 命令逐条执行
3. 可补充额外 grep（基于已有结果中的线索）
4. 提取推荐、证据等级、来源信息
5. 生成共识/差异分析
6. 写入 rag_batch_NNN.json
7. 报告进度

完成后执行 orchestration_plan.json 中 next_steps 的命令。
```

### 2.4 批次间上下文重置机制

在每个 `batch_NNN_prompt.md` 开头嵌入显式上下文重置指令：

```markdown
<CONTEXT_RESET>
请忽略此消息之前的所有检索结果和患者信息。
以下是一个全新的、独立的批次任务，从零开始处理。
不得引用或参考任何其他批次的结果。
不得使用"同上"、"与前面类似"、"参考前述"等表述。
```

> "在开始处理新批次前，先读取该批次的 prompt 文件。prompt 开头的 CONTEXT_RESET 指令要求你从零开始——不引用前批的任何内容。"

### 2.5 扩展 Forbidden Actions

新增：
- Never spawn subagents for batch processing
- Never write custom scripts to replace batch_pipeline.py
- Never process batches in parallel
- Never store output outside orchestration_plan.json 指定的目录
- Never skip any grep command from the prompt
- Never skip any org's search results
- Never ignore patient clinical features in search

## 设计决策说明

### 为什么不用静态配置文件？
用户批注明确要求"疾病的映射需要根据已配置指南来自动拓展生成，不要让用户配置"。知识库的 `data_structure.md` 已包含高度结构化的映射信息（组织列表、文件清单、临床问题→指南矩阵），可在 orchestrate 运行时自动解析。

### 为什么"全特征提取"由脚本+LLM 协作完成？
脚本从患者 26 个字段规则化提取基础关键词（确定性），LLM 可在 prompt 允许的空间内补充脚本未覆盖的关键词（灵活性）。兼顾确定性和智能。

### 为什么使用显式 `<CONTEXT_RESET>` 而非自动上下文管理？
每个 batch prompt 开头嵌入重置指令是当前架构下最可行的方案——比"写即释放"更强的文本约束，比"新会话"更低的用户操作负担。

### 双层防御策略
脚本层面：减少 LLM 自由度（路径、命令、组织覆盖都预定义）
HARD_CONSTRAINT 层面：防止 LLM 在剩余自由度内偏离（如用 subagent、跳过命令）
核心目标：提高不同模型输出结果的均质性，降低 LLM 智能水平对输出质量的影响。
