# v2.3 反偷懒方案：结构性执行证据 + 后验验证

## 问题背景

LLM 在 OpenCode 中执行批次检索时，处理 2 批后开始"偷懒"——合理化跳过 grep 执行，用前序批次结果编造后续批次。现有文字规则（`CONTEXT_RESET`、`MANDATORY_RULES`、`HARD_CONSTRAINT`）无法阻止。

**根因**：输出 JSON 不要求执行证据，验证无法检测 grep 是否真正执行。

**核心思路**：规则无效 → 改为要求可验证的执行证据。信息论层面的防御：不执行 grep 就无法获得正确的 snippet。

## 四层防御

| 层 | 名称 | 机制 |
|----|------|------|
| L1 | 执行证据 schema | 输出 JSON 新增 `execution_log` 字段，记录每条 grep 的 match_count + first_match_snippet |
| L2 | prompt 结构硬化 | grep 命令分配 CMD-ID，每患者末尾插入检查点，要求完整性声明 |
| L3 | verify-batch 子命令 | merge 前验证：命令覆盖率 + snippet 在知识库文件中的真实性 |
| L4 | 批次深度衰减检测 | validate 增强：检测后续批次证据量是否显著低于前序批次 |

## 修改文件

### 1. `scripts/batch_pipeline.py` — `generate_batch_prompt()` (L1+L2)

**grep 命令编号**：每条 grep 分配 `CMD-P{n}-{org}-{seq}` ID
```
CMD-P001-NCCN-01: grep -n -i "HER2\|曲妥珠" /path/NCCN/extracted/*.txt
  → 记录: match_count, first_match_snippet (≥30字)
```

**每患者末尾插入检查点**：
```
#### 检查点 [患者 1]
确认以上 N 条 grep 全部执行完毕，记录到 execution_log。
```

**输出 schema 扩展**：每个 `guideline_results` 条目新增：
```json
"execution_log": [
  {"cmd_id": "CMD-P001-NCCN-01", "match_count": 14, "first_match_snippet": "...≥30字..."}
]
```

患者级别新增 `execution_summary`:
```json
"execution_summary": {
  "total_commands_in_prompt": 12,
  "total_commands_executed": 12,
  "commands_with_zero_matches": ["CMD-P001-JGCA-02"]
}
```

### 2. `scripts/batch_pipeline.py` — 新增 `cmd_verify_batch()` (L3)

```bash
python scripts/batch_pipeline.py verify-batch --input-dir Output/batches/ [--kb-root /path]
```

验证逻辑：
1. **命令覆盖率**：从 prompt 解析 CMD-ID 列表，与 JSON execution_log 比对 → 缺失报 ERROR
2. **计数一致性**：`total_commands_in_prompt` vs prompt 实际命令数 → 不一致报 ERROR
3. **snippet 真实性**（需 `--kb-root`）：在声称的 source_file 中验证 snippet 是否存在 → 不存在报 ERROR
4. **空匹配 + 有推荐矛盾**：match_count=0 但有具体推荐内容 → WARNING

辅助函数：
- `_parse_prompt_commands(prompt_text)` → `list[dict]`：正则提取 CMD-ID + 命令
- `_verify_snippet(snippet, source_file, kb_root)` → `bool`：规范化空白后子串匹配

### 3. `scripts/batch_pipeline.py` — 增强 `cmd_validate()` (L4)

新增 `_check_batch_depth_decay(results)`:
- 按 batch_source 分组统计每批次的 total_matches、avg_snippet_length
- 后半段平均深度 < 前半段 40% → 报 WARNING "批次深度衰减"

### 4. `SKILL.md` — 更新执行流程

Step 3 → Step 4 之间插入 verify-batch：
```
orchestrate → LLM执行 → verify-batch → merge → validate → generate
                              ↓ FAIL
                        重新执行失败批次
```

### 5. 测试

- `tests/test_verify_batch.py`（新建）：prompt 解析、snippet 验证、覆盖率检查、PASS/FAIL 场景
- `tests/test_validate_enhance.py`（扩展）：批次深度衰减检测
- `tests/test_prompt.py`（扩展）：验证新 prompt 包含 CMD-ID 和检查点

## 实施顺序

1. **Phase 1**: 修改 `generate_batch_prompt()` — CMD-ID + 检查点 + execution_log schema
2. **Phase 2**: 新增 `cmd_verify_batch()` + 辅助函数 + argparse 注册
3. **Phase 3**: 增强 `cmd_validate()` — 批次深度衰减
4. **Phase 4**: 更新 SKILL.md 流程
5. **Phase 5**: 测试

## 验证方式

```bash
# 1. 运行现有测试确认无回归
python -m pytest tests/ -v

# 2. 运行 orchestrate 生成新格式 prompt，检查 CMD-ID 和检查点
python scripts/batch_pipeline.py orchestrate --patients Output/patients.json --output-dir Output/batches --batch-size 5
cat Output/batches/batch_001_prompt.md | grep "CMD-"

# 3. 模拟 verify-batch 对现有 batch 文件（预期 FAIL，因旧格式无 execution_log）
python scripts/batch_pipeline.py verify-batch --input-dir Output/batches/
```
