## 决策清单

### D1: scan_knowledge_base() 解析策略

**决策**: 正则匹配表头定位表格，fallback 到目录枚举

- 锚定 `templates/data_structure_root.md` 和 `templates/data_structure_org.md` 的模板格式
- 用正则匹配表头行（如 `| 指南 |`、`| 文件名 |`）来定位表格，而非按行号硬编码
- 解析失败时 fallback 到 `ls guidelines/*/extracted/*.txt` 枚举
- fallback 时打印 WARNING

### D2: extract_patient_features() narrative 格式处理

**决策**: 正则扫描 + confidence 标记

- 对 narrative 文本用预定义关键词字典做正则匹配，覆盖 9 个临床维度
- 统计命中维度数：≤2 个维度命中时标记 `confidence: "low"`
- confidence: low 时在 prompt 中提示 LLM "该患者信息稀疏，请从叙述中补充推断关键词"
- 理由: 成本极低（几行代码），但让 LLM 知道需要主动补充

### D3: prompt 大小超限处理

**决策**: 自动拆分子批次

- `generate_batch_prompt()` 估算生成的 prompt token 数
- 超限时将该批次拆成更小的子批次（如 5→3→2），递归直到 prompt 在限制内
- 用户无需干预
- 理由: orchestrate 是预处理步骤，自动拆分的复杂度可控

### D4: grep 关键词转义

**决策**: 全部转义 + 自动生成去括号变体

- `escape_grep_keyword(kw)` 统一转义 `[`, `]`, `(`, `)`, `.`, `*` 等 grep 特殊字符
- 如果原始关键词含括号，额外生成去括号版本（核心部分用 `.*` 连接）
  - 例: `CPS(≥10)` → 既搜 `CPS\(≥10\)` 也搜 `CPS.*10`
- 两个版本用 `\|` 合并到同一条 grep 命令中
- 理由: 脚本层面保证覆盖率，不依赖 LLM 智能水平（符合 v2.2 核心设计目标）

### D6: prompt token 阈值

**决策**: 默认 80K tokens + `--max-prompt-tokens` 参数可调

- 80K tokens ≈ 320K 字符
- 目标模型最低门槛为 128K 上下文，80K prompt 留 48K 响应空间
- 用户可通过 `--max-prompt-tokens 30000` 等方式适配较小模型
- 理由: 能执行此任务的模型至少 128K 上下文，保守默认浪费批次数

### D7: Output 并发保护

**决策**: 默认路径 + 检测已有 plan 提示续跑或指定新目录

- 默认 `--output-dir Output/batches/`
- 启动时检测目录是否已有 `orchestration_plan.json`
  - 有且含未完成批次 → 提示用户：续跑 checkpoint
  - 有且全部完成 → 提示用户：指定新 `--output-dir` 或确认覆盖
  - 无 → 正常创建
- 理由: 与 checkpoint 恢复机制自然衔接，next_steps 命令路径确定

### D9: validate 跨批次相似度检测

**决策**: 字符 bigram Jaccard 相似度，阈值 0.8

- 比较对象: 不同批次中不同患者的 `recommendation` 文本
- 方法: 对文本生成字符 bigram 集合，计算 Jaccard 相似度
- 阈值: > 0.8 报 WARNING（疑似上下文污染）
- 理由: 零依赖、对中文天然适用、bigram 粒度足够捕捉整段复制
