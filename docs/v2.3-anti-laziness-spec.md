# v2.3 Spec: 结构性执行证据与后验验证

> 状态: APPROVED
> 日期: 2026-03-25
> 版本: v2.3

---

## 1. 问题定义

**现象**: LLM (ZhipuAI/OpenCode) 执行批次检索时，处理 2 批（10 位患者）后开始偷懒——合理化跳过 grep 执行，用前序批次结果编造后续批次内容。

**观测到的偷懒模式**:
- "由于 token 限制和时间考虑，我需要更高效地处理剩余批次"
- "基于之前检索的指南信息，为剩余批次生成结果"

**环境**: 5 人/批 × 6 批，偷懒从第 3 批开始。

**根因**: 输出 JSON 不要求执行证据，后验验证无法检测 grep 是否真正执行。现有文字约束（`CONTEXT_RESET`、`MANDATORY_RULES`、`HARD_CONSTRAINT`）均为声明式规则，LLM 可选择性忽略。

**核心策略**: 从"声明式禁止"转向"结构性证据要求"。不执行 grep 就无法获得正确的 snippet——信息论层面的防伪。

---

## 2. 四层防御总览

| 层 | 名称 | 机制 | 检测时机 |
|----|------|------|---------|
| L1 | 执行证据 schema | 输出 JSON 要求 `execution_log` 字段 | LLM 生成时 |
| L2 | prompt 结构硬化 | CMD-ID 编号 + 检查点 + 完整性声明 | LLM 生成时 |
| L3 | verify-batch 子命令 | 命令覆盖率 + snippet 真实性校验 | merge 前 |
| L4 | 批次深度衰减检测 | 跨批次统计异常检测 | validate 时 |

工作流变化:
```
当前: orchestrate → LLM执行 → merge → validate → generate
新增: orchestrate → LLM执行 → verify-batch → merge → validate → generate
                                    ↓ FAIL
                              重新执行失败批次
```

## 3. L1: 输出 JSON Schema 扩展

### 3.1 `guideline_results` 条目新增 `execution_log`

```json
{
  "guideline": "NCCN",
  "version": "2026.V2",
  "recommendation": "新辅助治疗后应进行再分期评估...",
  "evidence_level": "Category 1",
  "source_file": "NCCN_GastricCancer_2026.V2_EN.txt",
  "source_lines": "234-267",
  "execution_log": [
    {
      "cmd_id": "CMD-P001-NCCN-01",
      "match_count": 14,
      "first_match_snippet": "HER2-positive: trastuzumab + chemotherapy is recommended as first-line"
    },
    {
      "cmd_id": "CMD-P001-NCCN-02",
      "match_count": 8,
      "first_match_snippet": "Neoadjuvant therapy: perioperative FLOT or SOX regimen"
    }
  ]
}
```

### 3.2 患者级别新增 `execution_summary`

```json
{
  "patient_id": "P001",
  "patient_name": "张三",
  "execution_summary": {
    "total_commands_in_prompt": 12,
    "total_commands_executed": 12,
    "commands_with_zero_matches": ["CMD-P001-JGCA-02"]
  },
  "clinical_questions": [...]
}
```

### 3.3 字段规则

| 字段 | 类型 | 约束 |
|------|------|------|
| `cmd_id` | string | 格式 `CMD-P{nnn}-{ORG}-{nn}`，与 prompt 中的 ID 精确对应 |
| `match_count` | int | grep 返回的匹配行数，0 表示无匹配 |
| `first_match_snippet` | string | 第一个匹配行的文本片段，≥30 字符 |
| `total_commands_in_prompt` | int | 必须等于 prompt 中 CMD-* 的总数 |
| `total_commands_executed` | int | 必须等于 `total_commands_in_prompt` |
| `commands_with_zero_matches` | list[str] | 列出 match_count=0 的 CMD-ID |

## 4. L2: Prompt 结构硬化

修改 `generate_batch_prompt()` (batch_pipeline.py:651-733)。

### 4.1 grep 命令编号

当前:
```
**NCCN（必须）:**
1. grep -n -i "HER2\|曲妥珠" /path/NCCN/extracted/*.txt
```

新:
```
**NCCN（必须）:**
CMD-P001-NCCN-01: grep -n -i "HER2\|曲妥珠" /path/NCCN/extracted/*.txt
  → 记录到 execution_log: {cmd_id, match_count, first_match_snippet (≥30字)}
CMD-P001-NCCN-02: grep -n -i "新辅助\|neoadjuvant" /path/NCCN/extracted/*.txt
  → 记录到 execution_log: {cmd_id, match_count, first_match_snippet (≥30字)}
```

CMD-ID 格式: `CMD-P{patient_index:03d}-{org_name}-{seq:02d}`

### 4.2 每患者检查点

在每位患者的 grep 命令列表之后插入:
```
#### ✅ 检查点 [患者 {n}: {name}]
确认以上 {cmd_count} 条 grep 命令全部执行完毕。
在 JSON 输出中填写该患者的 execution_summary:
  total_commands_in_prompt: {cmd_count}
  total_commands_executed: <实际执行数，必须等于 {cmd_count}>
  commands_with_zero_matches: [<列出 match_count=0 的 CMD-ID>]
```

### 4.3 输出要求嵌入完整 JSON schema 示例

prompt 末尾的"输出要求"替换为包含 `execution_log` 和 `execution_summary` 的完整 JSON 示例。

### 4.4 收紧补充检索措辞

当前 (line 723-725):
```
如果在上述 grep 结果中发现需要进一步深入的线索，
可补充执行额外的 grep 命令（限同一知识库范围内）。
```

改为:
```
所有必须命令（CMD-*）执行完毕且记录到 execution_log 后，
可补充执行额外 grep 命令。补充命令不需要 CMD-ID。
```

## 5. L3: verify-batch 子命令

### 5.1 接口

```bash
python scripts/batch_pipeline.py verify-batch \
  --input-dir Output/batches/ \
  --kb-root /path/to/kb    # 可选，启用 snippet 真实性校验
```

Exit code: 0 = 全部 PASS, 1 = 存在 FAIL。

### 5.2 验证逻辑

#### V1: 命令覆盖率 (ERROR)

从 `batch_NNN_prompt.md` 解析所有 `CMD-*` ID 列表，与对应 `rag_batch_NNN.json` 中所有 `execution_log[].cmd_id` 比对。缺失的 CMD-ID 报 ERROR。

#### V2: 计数一致性 (ERROR)

`execution_summary.total_commands_in_prompt` 必须等于 prompt 中实际 CMD-* 数量。不一致报 ERROR。

#### V3: snippet 真实性 (ERROR, 需 `--kb-root`)

对每个 `first_match_snippet`，在知识库中定位声称的源文件，验证 snippet 是否为文件内容的子串（空白规范化后匹配）。

查找路径: 遍历 `kb_root/*/extracted/` 目录下匹配 `source_file` 的文件。

不存在则报 ERROR——强烈暗示结果伪造。

#### V4: 空匹配矛盾 (WARNING)

如果某 CMD-ID 的 `match_count=0`，但该 org 在 `guideline_results` 中存在具体推荐内容（`recommendation` 长度 > 50），报 WARNING。


## 6. L4: 批次深度衰减检测

增强 `cmd_validate()` (batch_pipeline.py)。

### 6.1 新增 `_check_batch_depth_decay(results)`

```python
def _check_batch_depth_decay(results: list[dict]) -> list[str]:
    """检测后续批次执行深度是否显著衰减。"""
```

逻辑:
1. 按 `batch_source` 分组，统计每批次:
   - `total_matches`: 所有 `execution_log[].match_count` 之和
   - `avg_snippet_len`: 所有 `first_match_snippet` 的平均长度
   - `cmd_count`: 已记录的 CMD-ID 总数
2. 按批次序号排序
3. 检测: 后半段平均 `total_matches` < 前半段 40% → WARNING
4. 检测: 连续 3 批单调递减 → WARNING

### 6.2 集成到 `cmd_validate()`

在现有的 `_check_cross_batch_similarity()` 和 `_check_org_coverage()` 之后调用。

## 7. SKILL.md 更新

### 7.1 执行流程

Step 3 (Execute Retrieval) 后、Step 4 (Merge) 前插入:

```bash
# 验证批次执行证据
python scripts/batch_pipeline.py verify-batch \
  --input-dir Output/batches/ --kb-root $KB_ROOT
```

如果 verify-batch 报 FAIL:
- 提示用户哪些批次需要重新执行
- 删除失败批次的 `rag_batch_NNN.json`
- 重新执行对应的 `batch_NNN_prompt.md`

### 7.2 输出 JSON schema 文档

更新 schema 说明，包含 `execution_log` 和 `execution_summary` 字段。

### 7.3 Forbidden Actions

新增:
- **Never omit execution_log** from guideline_results output
- **Never omit execution_summary** from patient results
- **Never fabricate first_match_snippet** — it must be the actual text from grep output
